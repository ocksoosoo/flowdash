## 3. 프로젝트 구조 및 아키텍처

### 3-1. 디렉터리 구조
```
flowdash/
├─ README.md
├─ index.html
└─ cssfolder/
   ├── reset.css
   ├── layout.css
   ├── variables.css
   ├── theme.css
   ├── components.css
   └── responsive.css
└─ jsfolder/
   ├─ main.js
   ├─ storage.js
   ├─ header.js
   ├─ stats.js
   ├─ controls.js
   ├─ board.js
   ├─ todo.js
   └─ modal.js
```

프로젝트의 각 JS 파일은 단일 책임 원칙(SRP)에 따라 역할을 명확히 분리하여 설계 및 구현함

*   **`main.js`**: 애플리케이션의 진입점 및 각 모듈의 초기화 함수(`init`)를 일괄 실행하고 관리하는 파일
*   **`storage.js`**: 로컬 스토리지(LocalStorage) 관련 데이터 조회, 저장, 삭제 등 데이터 영속성 관리를 전담하는 파일
*   **`header.js`**: 사용자 맞춤형 인사말, 오늘 날짜 출력, 프로필 배경 색상 및 테마 토글, 닉네임 수정 등 상단 헤더 영역의 UI 및 데이터 상태를 제어하는 파일
*   **`stats.js`**: 전체 할 일 데이터에 따른 원형 그래프 렌더링 및 대시보드 통계 수치를 계산하고 업데이트하는 파일
*   **`controls.js`**: 할 일 목록의 검색, 정렬, 필터링 등 칸반 보드 전체 데이터의 흐름과 조회 방식을 제어하는 유틸리티 파일
*   **`board.js`**: 칸반 보드의 컬럼(To-Do, In-Progress, Done 등) 렌더링, 상태 변경 이벤트를 처리하는 파일
*   **`todo.js`**: 개별 할 일(Todo Item) 객체의 생성, 수정, 삭제(CRUD) 비즈니스 로직과 세부 상태 관리를 전담하는 파일
*   **`modal.js`**: 할 일 추가/수정 시 띄워지는 팝업창(모달)의 노출, 입력 폼 제어, 유효성 검사 및 닫기 이벤트를 관리하는 파일

### 3-3. 데이터 흐름 (Data Flow)

#### 1) 할 일 생성 (Create)
모달 등록 (`modal.js`) 
→ `createTodo()` 데이터 추가 (`todo.js`) 
→ `saveTodos()` 스토리지 저장 (`storage.js`) 
→ `refreshBoardWithFilter()` 호출 (`board.js`)
  - `getFilteredTodos()` 필터/정렬 적용 (`controls.js`)
  -  `initStatistics()` 대시보드 및 달성률 갱신 (`stats.js`)
  - `renderBoard()` 칸반 카드 출력 (`board.js`)

---

#### 2) 할 일 수정 (Update)
수정 버튼 클릭 (`board.js`) 
→ `openModal()` 기존 데이터 바인딩 (`modal.js`) 
→ 내용 수정 및 완료 클릭 
→ `updateTodo()` 데이터 및 수정일 갱신 (`todo.js`) 
→ `saveTodos()` 스토리지 저장 (`storage.js`) 
→ `refreshBoardWithFilter()` 실시간 UI 및 통계 갱신 (`board.js`)

---

#### 3) 할 일 삭제 (Delete)
삭제 클릭 및 확인 (`board.js`) 
→ `deleteTodo()` 데이터 제거 (`todo.js`) 
→ `saveTodos()` 스토리지 저장 (`storage.js`) 
→ `refreshBoardWithFilter()` 실시간 UI 및 통계 갱신 (`board.js`)

---

#### 4) 검색 및 필터/정렬 (Filter & Sort)
컨트롤러 입력 (`controls.js`) 
→ `filterState` 상태 변경 (`controls.js`) 
→ `getFilteredTodos()` 조건별 가공 (`controls.js`) 
→ `renderBoard()` 필터링된 보드 렌더링 (`board.js`)

## 6. 트러블슈팅(Troubleshooting)

### 6-1. 모달(Modal) 컴포넌트 렌더링 및 이벤트 바인딩 오류
* **증상**: 할 일 추가/수정 및 초기화 모달(Modal)이 화면에 정상적으로 출력되지 않는 현상 발생
* **원인**: 
  * `modal.js` 내 동일 이름의 중복 변수 선언 및 이벤트 리스너 누락
  * 진입점인 `main.js`에 모달 초기화 함수(`initModal`, `initResetModal`) 연동 누락 등 복합적 에러 발생
* **해결**: 
  * 팀원들이 각자 로컬 환경에서 디버깅 후 정상 작동 코드를 공유 및 교차 검증
  * 불필요하게 쪼개진 모달 관련 변수 및 리스너 등록 프로세스를 통합 정돈하여 해결
* **회고**: 
  * 모듈화(JS 파일 분리)로 인해 여러 파일을 동시에 추적해야 하는 번거로움은 있었으나, 에러 발생 모듈을 신속하게 특정할 수 있어 디버깅 효율이 높음을 확인
  * 브라우저 개발자 도구의 콘솔을 적극 활용하여 `ReferenceError`를 신속하게 추적 및 조치 완료


### 6-2. 웹 폰트 디코딩 실패로 인한 브라우저 과부하 및 지연 현상
* **증상**: 브라우저 콘솔창에 `sfntVersion, Failed to decode` 경고 메시지가 지속적으로 출력되며 렌더링 성능이 저하되는 현상
* **원인**: CSS에서 웹폰트 호출 시 최신 압축 포맷인 `.woff2` 규격을 브라우저가 구형 규격으로 잘못 해독하려고 시도하여 연쇄적인 병목 및 렉 유발
* **해결**: CSS 최상단에 웹 최적화 포맷인 `@import` 표준 호출 방식으로 코드를 전환하여 웹폰트를 정상 디코딩하고 브라우저 클릭 지연 현상 해결
* **회고**: 
  * 동작을 멈추게 하는 치명적인 빨간색 에러가 아닌 주황색 경고 메시지라도 서비스 전체의 성능 및 최적화에 중대한 영향을 미칠 수 있음을 인지
  * 웹 표준 포맷과 로딩 최적화 방식을 엄격하게 준수해야 함을 배움


### 6-3. 중복 변수 재선언으로 인한 자바스크립트 전체 동작 마비
* **증상**: 브라우저 로딩 시 어떠한 자바스크립트 기능도 동작하지 않고 전체 스크립트가 실행 불능 상태가 됨
* **원인**: 개별 코드를 병합하는 과정에서 최상단 변수 선언부 외에 파일 중하단 구역에 동일한 식별자의 변수(`const`)가 중복 재선언되어 구문 에러 발생
* **해결**: 충돌을 유발하던 미사용 중복 변수(`resetBtn`, `resetCancelBtn`, `saveModalCancelBtn` 등)를 식별 및 일괄 제거하여 정상 복구
* **회고**: 
  * 전역 마비와 같은 거대한 장애도 결국 오타나 변수 선언 중복 같은 기본적이고 사소한 문법 오류에서 비롯될 수 있음을 상기
  * 시스템 장애 발생 시 당황하지 않고 브라우저 디버거와 코드 흐름을 차근차근 검토하는 디버깅 태도의 중요성을 깨달음

- 디자인 시안과 일치하게 구현하는 데에 성공했고, 필수 구현 사항을 모두 충족함


### 7-2. 아쉬운 점
- 키보드 접근성을 챙기지 못함(특히 tab키 이동)
- 각자의 파트에 집중하느라 다른 파트를 이해하고 소화할 시간이 부족했음
- 초반 아이디어 회의 때 캘린더 추가에 대한 의견이 있었는데 난이도와 시간 부족의 이유로 구현하지 못한 점이 아쉬움

### 7-3. 다음에 개선할 점
- 웹 접근성을 전체적으로 챙기기
- 드래그 앤 드롭이나 아코디언 메뉴 등 다양한 형태의 UI 디자인과 기능 구현에 도전해봐도 좋을 것 같음
